# harden-spec

**Announce:** "harden-spec으로 spec을 적대적으로 압박해 굳힙니다."

## 핵심 원칙

brainstorming이 만든 **spec 초안을, plan/구현으로 넘어가기 전에 적대적으로 압박**해서, 늦게 발각되면 기능을 재설계·재구현하게 만드는 갭을 **지금** 메운다. 대상은 *작성된 spec 파일*이고, 결과물은 **더 단단해진 그 파일 자체**다(별도 문서 아님).

프리모템 태도로 본다: **"이 기능을 다 만들었는데 결국 재설계하게 됐다 — 무엇을 놓쳤었나?"**

## 언제 / 언제 아님

- **쓴다:** brainstorming으로 spec 초안이 있고 review-loop/plan 전에 굳히고 싶을 때. **또는 아직 파일은 없어도 이미 손에 쥔 계획·아이디어를 압박해 굳히고 싶을 때**(§입력 — 느슨한 계획 모드).
- **안 쓴다:**
  - 압박할 설계 자체가 백지다(계획조차 없다) → 먼저 `superpowers:brainstorming`.
  - 코드/구현의 결함을 찾는다 → `dev-workflow:review-loop`(codex).
  - 우호적 협업 설계를 하고 싶다 → `superpowers:brainstorming`.

**파이프라인 내 자리 (상보 관계):**

```
brainstorming → spec 초안 → [harden-spec] → review-loop(spec) → plan → 구현
  (협업·우호)              (적대·사람)        (codex·산출물)
```

harden-spec은 codex가 못 보는 것 — *사람만 아는 누락된 요구·숨은 가정·제품 지식* — 을 먼저 메워, 뒤따르는 review-loop가 진짜 결함에 집중하게 한다.

## 입력

- 대상 spec을 인자로 받는다(경로 또는 feature 슬러그).
- 생략 시 `docs/specs/`의 최신 파일 또는 현재 브랜치에 대응하는 spec을 찾아 **"이 spec을 압박합니다: `<path>` — 맞나요?"로 확인한 뒤** 시작한다.
- `docs/specs/`가 없으면 대상 문서를 사용자에게 묻는다.
- **느슨한 계획 모드 (파일 없는 입력):** 대상이 파일이 아니라 대화 중 제시된 계획·아이디어·초안이어도 된다. 이 경우:
  - 그 서술을 §절차 1의 "spec 주장"으로 삼고, repo 컨텍스트(CLAUDE.md·ADR·기존 spec)로 불변식·기결정만 보강한다 — 대상 파일이 없을 뿐 project-aware·프리모템은 동일.
  - 하드닝(§절차 4)은 반영할 파일이 없으므로, 종료 시 **"확정된 결정을 `docs/specs/`에 spec 초안으로 남길까요?"**를 제안한다 — 승인 시 초안 생성, 거절 시 결정·잔여 리스크 요약으로 종료.
  - 압박할 *계획 자체가 없는 백지*면 harden-spec이 아니라 `superpowers:brainstorming`이다.

## 절차 (5단계)

### 1. 로드 (context 우선)
대상 spec 본문 + 실행 repo의 `CLAUDE.md`·`AGENTS.md`·`docs/adr/`·`docs/specs/`를 읽어 세 바구니로 분류한다:
- **불변식/함정** — 이 repo가 반복적으로 밟는 규칙(CLAUDE.md에서 추출).
- **기결정** — ADR·spec `D<n>`·명시 규칙·이미 내려진 사용자 결정. **압박 대상에서 제외**(§기결정 가드).
- **spec의 주장** — 이 spec이 무엇을 하겠다는가(범위·정책·데이터·상태전이·AC).

repo에 위 문서가 없으면 범용 세트로 폴백하되, **일반 질문만 하지 말고** 최소한 spec 본문의 내부 정합성부터 판다.

### 2. 갭 스윕
§커버리지 축을 훑어 spec이 **침묵·모호·모순·미검증**인 지점을 갭 후보로 모은다. **rework 위험순으로 정렬**한다: 비가역(마이그레이션·데이터) → 교차모듈 파급 → 불변식 접촉 → 그 외. 사실로 확인 가능한 것은 코드·문서를 **직접 조회**해 갭 후보에서 제거한다(사용자에게 묻지 않는다).

### 3. 압박
위험 높은 갭부터 **한 번에 한 질문**, 매 질문에 **근거 있는 추천답**을 붙여 묻는다.
- 열린 갭/결정만 묻는다.
- 기결정은 건너뛴다(가드). 신규 갭이 기결정과 충돌하면 재론이 아니라 "이 기결정이 이 spec과 양립하는가"만 확인한다.
- **여러 질문을 한꺼번에 던지지 않는다** — 혼란을 유발하고 답을 얕게 만든다.

### 4. 하드닝 (제안 후 적용)
갭이 해소될 때마다 **spec에 넣을 구체 문구(diff)를 제안**하고, 승인되면 **파일에 반영**한다. 변경 단위를 작게 유지해 되돌리기 쉽게 한다. 결정·엣지케이스·실패 모드·범위 경계를 spec의 해당 섹션(결정사항·AC·비목표 등)에 실제로 기록한다.

### 5. 종료
§종료 산출 참조.

## 커버리지 축 (project-aware)

목록을 실행 repo의 CLAUDE.md에서 실제로 추출하고, 없거나 얕으면 이 범용 세트로 폴백한다.

| 축 | 묻는 것 |
|---|---|
| 완결성 | 명시 안 된 요구·엣지케이스·빈 상태·에러/실패 모드 |
| 불변식 준수 | repo 규칙 위반 소지(예: 권한 fail-closed, 트랜잭션·CAS·낙관락, 시각/타임존, enum 동기화) |
| 교차 영향 | 다른 모듈·기존 데이터·권한·마이그레이션에 미치는 파급 |
| 동시성·상태전이 | 트랜잭션 경계, 상태 전이표의 누락 전이, 경쟁 조건 |
| 비가역·롤백 | 마이그레이션 가역성, cutover, 롤백 계획 |
| 검증가능성 | 이 spec이 맞는지 어떻게 테스트/관측하나(AC가 검증 가능한가) |
| 범위 경계 | 무엇을 **안 하는지**가 명시됐나(scope creep 방지) |
| 숨은 가정 | 명시 안 된 가정이 뒤집히면 설계가 무너지나 |

## 기결정 가드 (review-loop 연동)

ADR·spec `D<n>`·명시 규칙·이미 내려진 사용자 결정은 **재론하지 않는다.** harden-spec은 기결정을 *전제로* 그 위의 갭만 판다.

| 합리화 | 실제 |
|---|---|
| "이 기결정이 틀린 것 같은데 다시 따져보자" | 기결정 번복은 harden-spec의 일이 아니다. 정말 위험하면 갭이 아니라 **DEFERRED로 남겨 사용자가 별도로 재론**하게 한다. 압박 흐름에서 재론하지 않는다. |
| "ADR을 못 봤으니 그냥 물어보자" | 먼저 `docs/adr`·spec `D<n>`을 **읽는다**. 거기 있으면 기결정이다. |
| "확실히 하려고 한 번 더 확인" | 기결정 재확인은 압박 예산 낭비다. 열린 갭에 쓴다. |

종료 시 확정된 결정·재론 금지 항목을 spec 안에 **`## 재논의 금지(기결정)` 블록**으로 남긴다. 이는 review-loop가 쓰는 규약(가드는 *리뷰 대상 문서 안*에 있어야 codex가 본다)과 같아, review-loop의 재지목을 선제적으로 줄인다.

## 종료 산출

세션 종료 시 대상 spec 파일에 다음이 있어야 한다:
1. 압박으로 확정된 결정·엣지케이스·실패 모드가 반영된 **실제 diff**.
2. **`## 재논의 금지(기결정)`** 블록(기결정 + 이번에 확정한 결정).
3. **`## 잔여 리스크 (DEFERRED)`** — 이번에 못 닫은 갭을 명시(은폐·무시 금지).

그다음:
- **커밋**: spec 파일만 **명시적으로 stage 후 커밋**한다.
  - `.git/index.lock`이 있으면 다른 세션이 git 사용 중 — 지우지 말고 기다린다.
  - `git add -A` 금지(무관한 untracked·다른 세션 작업 혼입 — 이 세션이 수정한 파일만 명시 stage).
  - 커밋 메시지에 AI 서명/도구 흔적 금지(글로벌 규칙).
- **정지**: 다음 단계(review-loop → plan)는 **자동 전환하지 않고** 권고만 한다. 단계 경계는 *새 세션 + `/clear`* 규약(핸드오프 후 사람이 시작).

## 하지 말 것 (red flags — 멈추고 자기점검)

- 여러 질문을 **한꺼번에** 던진다 → 한 번에 하나.
- **기결정(ADR·`D<n>`·사용자 결정)을 재론**한다 → 전제로 두고 갭만 판다.
- spec·CLAUDE.md를 **안 읽고 일반 질문**을 한다 → 먼저 로드해서 그 repo의 불변식으로 압박한다.
- 코드에서 확인 가능한 **사실을 사용자에게 묻는다** → 직접 조회한다.
- 압박이 끝나자 **다음 단계로 자동 전환**한다 → 정지하고 새 세션을 권고한다.
- 못 닫은 갭을 **조용히 넘긴다** → DEFERRED로 명시한다.
- `git add -A`로 커밋하거나 **AI 서명**을 넣는다 → 명시 stage + no-AI-trace.

## 인자

- (선택) 대상 spec 경로/슬러그. 생략 시 자동 탐색 + 확인.
- 그 외 인자는 두지 않는다(파일 하나를 읽으므로 git 기준 등 불필요).
